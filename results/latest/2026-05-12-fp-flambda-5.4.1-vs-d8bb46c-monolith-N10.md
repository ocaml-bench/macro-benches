## TL;DR

- **The 5-06 N=10 numbers held up across all 27 carried-over benchmarks.** Mean |Δ version-effect| across 108 paired cells (4 flag combos × 27 benches) is **0.44pp**; the largest single-cell swing is 2.8pp on `zarith_pi` baseline. Headline findings — owl_gc, alt_ergo_fill, jsoo, cpdf_*, menhir_sysver, ocamlc_self_compile, liq_parse_typecheck — all confirmed with version effects within ±1pp.
- **`liq_video_frames_pool` is neutral at default `M`.** The variant uses `LIQ_POOL=1` + `LIQ_TOUCH=full` — a refcounted-pool stub modelling ffmpeg's `AVFrame` / `av_frame_free` lifecycle and the toots/[ocaml#14533](https://github.com/ocaml/ocaml/issues/14533) reproducer. It sits at **+3% wall, +0% RSS, no GC-overhead drop** across the 4 flag combos — exactly the shape #14533 predicts at default `M`: refcounted-pool semantics give no regression *and* no improvement at `(M=44, o=120)`. The "free lunch" from #14533 needs the large-`M` regime to materialise.
- **The 4 new `lavyek_kv_{1d,2d,4d,8d}` benchmarks are quiet on the version axis.** Max version effect across all 16 (bench, flag) cells is **+3.8%** — none cross the 5% line. One diagnostic flag: `lavyek_kv_8d` allocates **12% more** under d8b (5.11 → 5.74 GB total), with 12% more minor collections to match. Worth a note but not a regression at the wall-time level.
- **`zarith_pi` baseline reverts to 0.0%** — the +2.8% from 5-06 N=10 was itself sampling, not a stable signal. Across three runs (N=3 monolith: +0.0%, N=10 monolith 5-06: +2.8%, N=10 monolith 5-12: +0.0%) the baseline cell is flip-flopping; the obelisk-vs-monolith microarchitecture story is back to "monolith does not see the page-fault regression at default `o`".
- **Five other ≥1.5pp shifts**, all within ±5% (i.e. noise band): `sedlex_tokenize -flambda` (-1.4% → +1.0%, sign flip), `alt_ergo_yyll` baseline (+1.4% → +3.2%), `owl_gc` baseline (-36.4% → -38.4%, gets slightly better), `liq_parse_typecheck -flambda` magnitude shrinks (-6.3% → -4.6%), `sedlex_tokenize` baseline (-4.7% → -3.2%). None invalidate prior conclusions.

## Findings

### 1. New `liq_video_frames_pool` — the toots/#14533 reproducer is neutral at default `M`

The old single `liq_video_frames` benchmark has been replaced by
`liq_video_frames_pool` — the refcounted-pool variant that reproduces toots'
[ocaml#14533](https://github.com/ocaml/ocaml/issues/14533) free-lunch shape:

- `LIQ_POOL=1` — refcounted-pool stub modelling ffmpeg's `AVFrame` /
  `av_frame_free`. Calls [`pool_alloc`](../benchmarks/liq-video-frames/pool_stubs.c)
  → `caml_alloc_custom_mem(ops, sizeof(intnat), size)` — registers `size`
  bytes with the pacer for accounting but allocates *no buffer*; the OCaml
  side touches a single shared Bigarray. Finalize is the default (refcount
  decrement, no `free`), so RSS is bounded by the shared buffer regardless of
  GC release cadence.
- `LIQ_TOUCH=full` — every-pixel write per frame, preserving the real-pipeline
  mutator cost.

| variant | POOL | TOUCH | base | -fp | -flambda | -fp-flambda | GC overhead 5.4 → d8b base | RSS Δ |
|---|---|---|---|---|---|---|---|---|
| **`_pool`** | **1** | **full** | **+3.0%** | **+6.5% ⚠️** | **-7.4% ✓** | **-1.0%** | **91.2% → 91.4%** | **+0.4%** |

`_pool` sits at 91% GC overhead on 5.4 — high pacer-pressure regime — and d8b
**does not** drop the overhead (91.2% → 91.4%); RSS stays flat (+0.4%). The
refcounted-pool semantics cap committed memory regardless of release cadence,
so the new pacer has nothing to gain on this workload at default `M`. Wall
sits at +3% / +6.5% / -7.4% / -1.0% — essentially neutral across flag combos.
The "free lunch" predicted by #14533 (d8b's lazier pacer = less CPU, same
RSS) needs the **large-`M` regime** to materialise; the 5-11 `(M, o)` sweep
used the *default* (non-pool) binary, so the refcounted variant has not yet
been swept where the free-lunch effect should be cleanest. See follow-up #3
below.

> **Noise floor for `_pool`:** d8b-flambda has a 27.1% within-cell wall spread
> (one slow draw among otherwise ~3.0s) — the -7.4% improvement should be read
> as "no regression" rather than as a real improvement at default `M`. d8b
> baseline spread is 13.5%. Medians are stable but cell-level claims under
> ±5% are inside their spread.

None of the 16 (bench, flag) cells cross the 5% line; the suite is well-behaved
on the version axis. RSS hovers at ~2 GB across all variants (these are big
WAL-style workloads) with d8b consistently 1.5-4.5% lower.

The single diagnostic worth flagging: **`lavyek_kv_8d` allocates 12% more under
d8b** (5.11 GB → 5.74 GB total heap), with 12% more minor collections
(3634 → 4088) to match. The wall does not visibly regress (+3.7%), so the
allocation amplification is partly absorbed by the GC pacer working harder
without hurting throughput. Same shape as `ocamlc_self_compile`'s 16% allocation
growth, just smaller in magnitude and on a wider working set. Worth checking
which Lavyek code path is responsible — likely the same area as the
ocamlc_self_compile allocation growth (typer / Hashtbl resize / Marshal).

> Within-cell wall spread for lavyek is moderate (`_8d` p90 ~7%; `_4d` p90 ~4%);
> the medians are stable but a 1pp version-effect signal would not survive
> N=10.

## Methodology notes

- **Median over mean** for all headline numbers; consistent with prior reports.
- **Spread** (max − min)/median per cell: mean across all 256 cells is ~2%,
  median ~1.5%. Cells with ≥10% spread:
  - `liq_video_frames_pool` d8b-fl: 27.1% (one slow draw)
  - `eio_fiber_stream` d8b-fpfl: 13.3%
  - `liq_video_frames_pool` d8b: 13.5%
  - `coqc_corelib_stress` d8b-fp: 11.9% (I/O-bound control)
  The new `_pool` variant is noisier than the original single `liq_video_frames`
  benchmark was (which sat at ~3% spread); the small (sub-±5%) `_pool` cells
  should be treated with this in mind.
- **No formal statistical test.** N=10 stable for medians, not enough for proper
  significance testing (Le Boudec 2010 / Chen 2012 want ~30 per cell).
- **Runner data integrity.** All 2560 invocations captured cleanly across 32
  benchmarks × 8 variants × 10 invocations. No tmpdir-leak fallout
  (running-ng@6ee044b is current). `/tmp` had ample free space throughout the
  run.
- **No OOMs, timeouts (other than the deliberate 15s timelimit on
  `alt_ergo_unsat_smt2`), or build failures.**

## Suggested follow-ups

1. **Run the (M, o) sweep on `liq_video_frames_pool`.** At default `M=44` it

   *at large `M`* d8b's lazier custom-block pacer should produce a CPU win
